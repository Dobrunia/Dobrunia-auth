# Dobrunia Auth — подключение к продукту

**Dobrunia Auth** — готовый сервис единого входа для приложений экосистемы Dobrunia. Вы работаете с базовым URL API и самостоятельно регистрируете приложение-клиент через защищённый API.

Для текущего production-контура:

- **API:** `https://dobrunia-auth.na4u.ru`
- **Auth Web UI:** `https://auth.dobrunia.guru`

При самостоятельном размещении Auth Web настройте SPA fallback в nginx: `try_files $uri $uri/ /index.html;`. Пример находится в `client/nginx.conf.example`. OAuth bridge дополнительно использует корневой bootstrap URL, поэтому вход не зависит от прямой доступности `/oauth-bridge` на статическом сервере.

Ниже — что нужно для интеграции в свой проект (веб, мобильный клиент + бэкенд).

---

## Как зарегистрировать приложение-клиент

Рекомендуемый способ: откройте Auth Web, зарегистрируйтесь или войдите, затем перейдите на страницу **`/clients`** («Мои приложения»). Форма создаст приложение от имени текущего пользователя, а Web UI самостоятельно обновит истекший access token. Копировать пользовательский токен вручную не требуется.

Для автоматизации доступен тот же flow через API. Сначала войдите в существующий аккаунт Dobrunia и получите `accessToken`. Для локальной установки можно использовать системный клиент `dobrunia-auth-web`:

```http
POST {BASE}/auth/login
Content-Type: application/json

{
  "email": "developer@example.com",
  "password": "********",
  "clientId": "dobrunia-auth-web"
}
```

Если аккаунта ещё нет, отправьте такое же тело в `POST {BASE}/auth/register`.

Затем зарегистрируйте своё приложение:

```http
POST {BASE}/clients
Authorization: Bearer <accessToken>
Content-Type: application/json

{
  "name": "My Local App",
  "slug": "my-local-app",
  "description": "Локальное приложение для разработки",
  "baseUrl": "http://localhost:4173",
  "redirectUris": [
    "http://localhost:4173/oauth/callback"
  ]
}
```

Ответ `201` содержит `client.id` и `client.slug`. Любое из этих значений можно передавать как `client_id` в OAuth URL. `client_id` постоянный и именно его следует хранить в конфигурации интегрируемого приложения. Access token владельца нужен только для управления клиентами и не должен попадать в `.env` приложения. `slug` уникален; разрешённые callback URL сравниваются точным совпадением. В production разрешены только HTTPS URL, а HTTP допускается только для `localhost`, `127.0.0.1` и `[::1]`.

Секрет клиенту не выдаётся: текущая интеграция использует публичный OAuth-клиент и authorization code. Регистрация клиента не меняет `CORS_ORIGINS`; для запросов к Auth API напрямую из браузера origin по-прежнему должен быть разрешён в конфигурации сервера. Обмен OAuth-кода рекомендуется выполнять на бэкенде приложения.

На странице `/clients` можно редактировать настройки, отключать и удалять приложение, видеть количество активных пользователей/сессий и завершать отдельные сессии. Те же операции доступны через `PATCH /clients/:id`, `DELETE /clients/:id` и `/clients/:id/management/sessions`. Удаление необратимо и каскадно удаляет сессии, refresh tokens и OAuth-коды приложения.

## Что вы получаете при подключении

| Данные | Зачем |
|--------|--------|
| **Базовый URL API** | Все запросы идут на этот origin: `{BASE}/auth/...`, `{BASE}/oauth/...` |
| **`client_id`** | Идентификатор вашего приложения в Auth: **UUID** или короткий **`slug`** (строка вроде `my-shop-web`) |
| **Разрешённые `redirect_uri`** | Список точных HTTPS-URL возврата после входа; для loopback-разработки разрешён HTTP |

Дополнительно в профиле приложения у сервиса могут быть заданы **название**, **логотип** и **сайт** — они используются на экране входа и в ответах API там, где отдаётся информация о клиенте.

Если ваш фронт ходит в API из браузера, **origin** сайта должен быть внесён в `CORS_ORIGINS` на стороне Dobrunia Auth. Это отдельная настройка от `redirect_uri`.

## Два способа встроить вход

### 1. Рекомендуемый: OAuth 2.0 Authorization Code

Подходит, когда пользователь нажимает **«Войти через Dobrunia»** и вы не хотите собирать пароль Dobrunia на своих страницах.

**Шаг 1 — ссылка с вашей кнопки** (открыть в том же окне или через `location.assign`):

```text
{BASE}/oauth/authorize?response_type=code&client_id={ВАШ_CLIENT_ID}&redirect_uri={URL_ENCODED_CALLBACK}&state={СЛУЧАЙНАЯ_СТРОКА}
```

- `redirect_uri` — один из заранее согласованных URL, **символ в символ** как в списке у провайдера (включая слэш в конце, если он там есть).
- `state` — сгенерируйте до редиректа, сохраните (session / sessionStorage), на callback **сравните** с пришедшим — защита от CSRF.

**Шаг 2 — callback**  
Пользователь попадает на ваш URL с query: `?code=...&state=...`. Код одноразовый, короткоживущий.

Если пользователь уже вошёл в Auth Web, промежуточный bridge создаёт сессию вашего приложения без повторного ввода пароля. Cookie устанавливается top-level POST-переходом на API, а не cross-site fetch. При сетевом таймауте или неудачной установке cookie bridge показывает явную ошибку вместо бесконечного loader.

**Шаг 3 — обмен `code` на токены**  
Вызовите с **бэкенда** своего приложения (так безопаснее, чем из браузера):

```http
POST {BASE}/oauth/token
Content-Type: application/json

{
  "grant_type": "authorization_code",
  "code": "<из query>",
  "redirect_uri": "<тот же redirect_uri, что в шаге 1>",
  "client_id": "<ВАШ_CLIENT_ID>"
}
```

**Ответ при успехе** (JSON):

- `accessToken` — JWT; в запросах к API Dobrunia Auth: заголовок `Authorization: Bearer <accessToken>`
- `refreshToken` — непрозрачная строка для продления сессии (`POST /auth/refresh`) и выхода (`POST /auth/logout`)
- `user`: `id`, `email`
- `session`: `id`, `clientId`, `clientSlug`

Дальше вы сами решаете, как хранить токены у пользователя (httpOnly-куки своего домена, сессия сервера и т.д.).

---

### 2. Прямой вход по API (своя форма логина/регистрации)

Если договорённости с продуктом позволяют собирать **email и пароль** на вашей стороне:

- Регистрация: `POST {BASE}/auth/register`
- Вход: `POST {BASE}/auth/login`

Тело JSON:

```json
{
  "email": "user@example.com",
  "password": "********",
  "clientId": "ваш-slug-или-uuid"
}
```

(вместо `clientId` можно передать `clientSlug` — то же значение.)

Ответ совпадает по полям с успешным ответом обмена кода (токены + `user` + `session`).

## Профиль пользователя для UI (аватар, имя)

После входа можно запросить:

```http
GET {BASE}/auth/me
Authorization: Bearer <accessToken>
```

В ответе, среди прочего:

- **`user`**: `id`, `email`, `firstName`, `lastName`, `avatarUrl` — последние три могут быть `null`, пока не заполнены в учётной записи Dobrunia.
- **`session`**: текущая сессия и клиент (`clientName`, `clientSlug` и т.д.).

Используйте это для шапки сайта, аватарки и подписи пользователя.

## Продление сессии и выход

- **`POST {BASE}/auth/refresh`** — тело `{ "refreshToken": "..." }`; в ответе новая пара `accessToken` + `refreshToken` (старый refresh после успеха недействителен).
- **`POST {BASE}/auth/logout`** — тело `{ "refreshToken": "..." }`; завершает текущую сессию на стороне Auth. Локально токены всё равно удалите.

Пользователь может видеть и отзывать **все** свои входы в кабинете Dobrunia Auth; для своего приложения вам обычно достаточно описанных методов.

Завершенные сессии сразу скрываются из API и интерфейса. Сервер физически удаляет их фоновой job, по умолчанию раз в час. Однократную очистку для системного cron можно выполнить командой `cd server && npm run sessions:cleanup`.

---

## Ошибки и безопасность

- Успешные и ожидаемые ошибки приходят как JSON: `{ "error": { "message": "..." } }` с соответствующим HTTP-статусом.
- Не логируйте и не передавайте **refresh** и **code** третьим лицам; **access** живёт недолго — тоже не светите в публичных URL.
- Всегда проверяйте **`state`** на OAuth callback.
- Продакшен: только **HTTPS** для ваших `redirect_uri` и для обращения к API.
