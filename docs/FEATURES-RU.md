# Как работает Dobrunia Auth

Этот документ объясняет основные возможности сервиса без деталей реализации. Полный HTTP-контракт находится в [API-ENDPOINTS.md](API-ENDPOINTS.md).

## Что здесь является клиентом

Не путайте три сущности:

- **Пользователь** — человек с email и паролем. Один и тот же аккаунт может входить в разные приложения.
- **Клиент** — приложение, подключенное к Dobrunia Auth: сайт, SPA, мобильное приложение или backend. У него есть постоянный `client_id`, slug и список разрешенных callback URL.
- **Сессия** — конкретный вход пользователя в конкретное приложение с access и refresh token.

Отдельной роли «разработчик» пока нет: любой вошедший пользователь может зарегистрировать свои приложения.

## Основные возможности

| Задача | Куда обращаться |
| --- | --- |
| Создать аккаунт | Web UI `/register` или сервер `POST /auth/register` |
| Войти | Web UI `/login` или сервер `POST /auth/login` |
| Зарегистрировать приложение | Web UI `/clients` или сервер `POST /clients` |
| Посмотреть свои приложения | Web UI `/clients` или сервер `GET /clients` |
| Изменить или отключить приложение | Web UI `/clients` или сервер `PATCH /clients/:id` |
| Удалить приложение | Web UI `/clients` или сервер `DELETE /clients/:id` |
| Посмотреть пользователей своего приложения | Web UI `/clients` или сервер `GET /clients/:id/management/sessions` |
| Завершить сессию в своём приложении | Web UI `/clients` или сервер `DELETE /clients/:id/management/sessions/:sessionId` |
| Начать OAuth-вход | Сервер `GET /oauth/authorize` через redirect браузера |
| Обменять OAuth-код на токены | Сервер `POST /oauth/token`, желательно с backend приложения |
| Обновить токены пользователя | Сервер `POST /auth/refresh` |
| Получить профиль | Сервер `GET /auth/me` |
| Посмотреть или завершить входы | Web UI `/` или сервер `/sessions` |

Пути Web UI открываются в браузере на адресе auth-приложения. Серверные endpoints вызываются на базовом URL API.

## Flow подключения нового проекта

1. Владелец проекта открывает Auth Web, регистрируется через `/register` или входит через `/login`.
2. На странице `/clients` он создает приложение и задает точные callback URL.
3. Сервис выдает постоянный `client_id` (UUID; slug также принимается). В приложение добавляют, например:

```env
DOBRUNIA_AUTH_URL=https://dobrunia-auth.example
DOBRUNIA_CLIENT_ID=your-client-id
DOBRUNIA_REDIRECT_URI=https://app.example/auth/callback
```

4. Кнопка «Войти через Dobrunia» перенаправляет браузер на `GET /oauth/authorize` с `client_id`, `redirect_uri`, `response_type=code` и случайным `state`.
5. Если пользователь уже вошёл в Auth Web, bridge подтверждает его текущую сессию и создаёт отдельную сессию целевого приложения без повторного пароля. Скрытая top-level POST-форма открывает API как first-party страницу, API ставит httpOnly cookie и возвращает браузер в OAuth flow через `303`. Иначе открывается форма входа.
6. Dobrunia Auth возвращает браузер на callback приложения с одноразовым `code`.
7. Backend приложения проверяет `state` и отправляет `code` в `POST /oauth/token`.
8. Полученные access и refresh token относятся к вошедшему пользователю. Приложение хранит их в своей серверной сессии или защищенных httpOnly cookies и ротирует через `POST /auth/refresh`.

Bridge не может висеть бесконечно: сетевые запросы ограничены таймаутом, login-router не возвращает недействительную сессию обратно в bridge, а повторный возврат после неустановленной cookie останавливается с явной ошибкой.

## Управление своим приложением

На странице `/clients` владелец может менять название, slug, описание, сайт, логотип, callback URL и состояние приложения. Отключение через `isActive: false` сохраняет настройки, но запрещает новые входы. `client_id` остаётся неизменным.

Для каждого приложения показываются число активных сессий и число уникальных пользователей. Владелец может открыть список активных входов и завершить отдельную сессию. Это не даёт доступа к паролям или токенам пользователя.

Удаление приложения необратимо: вместе с ним база каскадно удаляет его сессии, refresh tokens и OAuth-коды. Для временной остановки безопаснее отключить приложение, а не удалять его.

## Что истекает, а что нет

- `client_id` постоянный, не является секретом и не требует обновления. Именно его хранит работающий проект.
- Access token пользователя по умолчанию живет 1 час; срок настраивается сервером.
- Refresh token по умолчанию живет 30 дней и заменяется новым при каждом refresh.
- Authorization code одноразовый и живет 10 минут.
- Access token владельца нужен только для управления клиентами. Web UI обновляет его автоматически; помещать его в `.env` проекта нельзя.

## Как удаляются завершенные сессии

При нажатии «Завершить сессию» сервер сразу отзывает refresh tokens и переводит сессию в `revoked`. Интерфейс удаляет только эту карточку без повторной загрузки остальных данных, а API больше не возвращает завершенную запись.

Физическая очистка БД выполняется фоновой job при старте сервера и затем раз в час. Интервал задается через `SESSION_CLEANUP_INTERVAL_MINUTES`. Для внешнего cron можно отключить встроенную job через `SESSION_CLEANUP_ENABLED=false` и запускать:

```bash
cd server && npm run sessions:cleanup
```

Удаление каскадное: вместе с сессией удаляются связанные refresh tokens и OAuth-коды.

## Важные ограничения

Текущая версия регистрирует **public OAuth clients**: `client_secret` не выдается, PKCE пока не реализован. Flow работоспособен благодаря точному `redirect_uri`, `state`, короткому одноразовому коду и обмену на backend, но перед открытой интеграцией недоверенных сторон рекомендуется добавить PKCE или confidential clients с безопасной ротацией секретов.

Регистрация приложения не меняет CORS-конфигурацию сервера. Если frontend вызывает API напрямую, его origin должен быть разрешен через `CORS_ORIGINS` или настроенный reflect-режим.
