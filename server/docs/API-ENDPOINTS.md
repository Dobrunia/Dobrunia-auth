# Endpoints

Краткое описание публичных HTTP-эндпоинтов сервера (как сейчас реализовано в коде). Базовый URL в разработке: `http://localhost:<PORT>` (порт из `PORT` в `.env`, по умолчанию `3000`).

Ошибки с известным статусом отдаются как JSON: `{ "error": { "message": "..." } }`. Неожиданные ошибки — `500` с тем же видом и текстом `Internal Server Error` (без деталей стека в теле).

---

## `GET /health`

### Flow

- **Зачем:** проверка, что процесс жив (оркестраторы, балансеры, CI).
- **Что делается:** контроллер вызывает сервис без обращения к БД; возвращается фиксированный `status: ok` и текущий `timestamp` в ISO.
- **Ответ:** `200`, тело `{ "status": "ok", "timestamp": "<ISO-8601>" }`.

---

## `POST /auth/register`

### Flow

- **Зачем:** создать глобального пользователя в auth-сервисе и сразу выдать сессию + токены для указанного **клиента** (приложения-потребителя), чтобы приложение могло считать пользователя «вошедшим».
- **Тело (JSON):** `email`, `password`, и обязательно один из идентификаторов клиента: `clientId` или `clientSlug` (строка может быть UUID клиента или slug, например `shop-web`). Email нормализуется (trim + lower case). Длина пароля задаётся константами в `validation.constants`.
- **Из запроса:** для сессии сохраняются `ip` (с учётом `trust proxy` и `X-Forwarded-For`) и `User-Agent`.
- **Валидация:** Zod; при ошибке — `400`.
- **Бизнес-логика (транзакция):**
  - резолв активного клиента по `clientId` / `clientSlug`;
  - проверка, что email ещё не занят;
  - хеш пароля (argon2id);
  - вставка пользователя;
  - создание сессии со статусом `active`, привязка к `user_id` и `client_id`;
  - генерация **opaque** refresh-токена, в БД пишется только **SHA-256** от него;
  - выпуск **access** JWT (HS256, срок из `ACCESS_TOKEN_EXPIRES_SEC`).
- **Успех:** `201`, тело: `user` (`id`, `email`), `session` (`id`, `clientId`, `clientSlug`), `accessToken`, `refreshToken` (сырой refresh только здесь; в БД — хеш).
- **Ошибки:** `400` (невалидное тело по Zod; клиент не найден или неактивен — сообщение `Unknown or inactive client`), `409` (email уже зарегистрирован или дубликат на уровне БД).

## `POST /auth/login`

### Flow

- **Зачем:** войти существующим пользователем в контексте конкретного клиента; выдать **новую** сессию и пару токенов (как при регистрации).
- **Тело:** то же, что у register (`email`, `password`, `clientId` или `clientSlug`).
- **Из запроса:** IP и User-Agent пишутся в новую сессию.
- **Валидация:** Zod → при ошибке `400`.
- **Бизнес-логика (транзакция):**
  - резолв активного клиента;
  - загрузка пользователя по email с полями для проверки пароля и `is_active`;
  - отказ без детализации, если: пользователя нет, аккаунт неактивен, нет `password_hash` (например, только OAuth), пароль не совпал — везде одно сообщение для клиента;
  - обновление `last_login_at` у пользователя;
  - новая сессия `active`, новый refresh (в БД — хеш), новый access JWT.
- **Успех:** `200`, формат тела как у register (`user`, `session`, `accessToken`, `refreshToken`).
- **Ошибки:** `400` (только ошибки Zod-валидации тела). Неизвестный/неактивный клиент, неверные учётные данные, неактивный пользователь, отсутствие пароля в БД — всё это отдаётся как **`401`** с текстом `Invalid email or password` (намеренно одно сообщение). Редкий конфликт на уровне БД при вставке — `409` `Session conflict`.

## `POST /auth/logout`

### Flow

- **Зачем:** завершить **текущую** сессию, связанную с выданным refresh-токеном (одно устройство / один вход).
- **Тело (JSON):** `refreshToken` — тот же непрозрачный токен, что вернули register/login (не хеш).
- **Логика (транзакция):**
  - считается SHA-256 от переданного refresh;
  - ищется **активная** запись в `refresh_tokens` (не отозвана, не просрочена);
  - если найдена — помечается отзыв refresh с причиной `logout`, сессия переводится в `revoked` с той же причиной;
  - если не найдена (неверный токен, истёк, уже logout) — **ничего не ломаем**: идемпотентное поведение.
- **Успех / «тихий» случай:** `204`, **пустое тело** (клиент всё равно удаляет токены локально).
- **Ошибки:** `400`, если тело не прошло валидацию (например, нет или пустой `refreshToken`).

## Связка с архитектурой продукта

- **Клиент** в БД — это приложение экосистемы; без `clientId`/`clientSlug` нельзя однозначно привязать сессию к проекту.
- **Сессия** хранится в auth-сервисе; downstream-приложение после обмена токенов может поднять свою серверную сессию отдельно.
- **Refresh** никогда не хранится в открытом виде в БД — только хеш; подтверждение владения на logout — снова хеш переданного значения.
